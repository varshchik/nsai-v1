# Архитектура

Этот документ — сжатый русский обзор архитектуры NSAI. Полная версия — [principles_ru.md](./principles_ru.md).

NSAI — символьная система для разбора, хранения и рассуждения над фактами, извлечёнными из текста на естественном языке. Принцип, лежащий в основе каждого решения ниже, один: **не формализуй то, что уже эмерджентно из базового цикла.** Когда предложенный механизм дублирует то, что и так производит обход графа, механизм отвергается.

## Базовый цикл

Сигнал → Framing (сжатие) → Поэтажный обход → Оценка согласованности.

Агент принимает сигнал, фреймит его относительно глобального графа, обходит этаж за этажом и выдаёт оценку согласованности запрошенного утверждения с перспективы стартовых узлов. Каждый этаж расширяет фронт ассоциаций на один шаг. На каждом этаже подсчитываются все участники подтянутых фактов — и предикаты, и аргументы.

Счёт — свойство обхода, не графа. Когда обход заканчивается, счёт исчезает. Следующий запрос пересобирает его с нуля.

Два режима:
- **`perceive`** — принять сигнал, записать факты в граф. Память меняется.
- **`reason`** — сгенерировать ответ через обход, без записей. Память read-only.

Граф мутирует только от внешнего сигнала через `perceive`. Внутреннее рассуждение никогда не модифицирует хранилище.

## Framing как сжатие, а не расширение

Сигнал не семя для расширения — он фильтр. Каждая ассоциация, подтянутая из графа, получает вес: сколько лемм сигнала одновременно её активируют. Ассоциация, которой коснулась одна лемма из четырёх — шум; ассоциация, которой коснулись все четыре — максимальное совпадение.

Порог: при N=1 фильтр отключён (пересекать нечего); при N≥2 — `max(2, ⌈N/2⌉)`. То есть как минимум две леммы сигнала должны одновременно активировать факт. Это отсеивает «галлюцинацию через популярные леммы»: одиночная лемма, часто встречающаяся в графе, больше не тащит в выборку любые факты с её участием.

Если пересечение пустое, система переходит в open-world: граф ничего совместного не знает, и запрос честно помечается как новая территория.

## Coherence score: прямые наблюдения и мосты

Согласованность складывается из двух независимых счётчиков, каждый в своих единицах:

- **`direct_score`** — число фактов с полным покрытием сигнала (`_cov = N`). Каждый — одно прямое наблюдение в графе.
- **`bridge_score`** — число цепочек, в которых не-query лемма является общим соседом ≥2 query-лемм из непересекающихся стартовых ветвей. Каждая — одно косвенное свидетельство.

Сила отдельной цепочки (`a_facts × b_facts`) — диагностический показатель моста, в общий score не входит. Иначе популярные узлы графа (типа «человек», «вещь») доминировали бы за счёт широких окрестностей, не за счёт релевантности к запросу.

## Поэтажный обход на примере

Вопрос: «Сократ смертен?» В графе: Сократ — человек; люди смертны.

- **Этаж 1.** От `сократ` — факт «является человеком». От `смертный` — факт «смертный → человек».
- **Этаж 2.** Обе леммы запроса достигают `человек` из непересекающихся ветвей.
- **Результат.** `человек` — *bridge*: до него дотянулись две query-леммы через непересекающиеся стартовые пути. Цепочка `сократ → человек → смертный` — побочный продукт обхода, не отдельный шаг вывода.

Силлогистическое рассуждение эмерджентно из механики обхода. Нет inference engine — есть только подсчёт достижимости.

Глубина обхода — настраиваемый параметр. Глубина 1 даёт только прямые связи; глубина 3 — цепочки рассуждения; глубже — растёт шум.

## Согласованность всегда перспективна

У одного объекта могут одновременно быть несколько утверждений с высокой согласованностью. «Стакан наполовину полон» и «стакан наполовину пуст» — оба дают высокий score: они активируют разные окрестности ассоциаций, и обе согласованы с графом. Это не противоречие, это две валидные перспективы.

Система не сходится к единственной истине. Она честно возвращает согласованность под тем углом, с которого подошёл вопрос.

## Сравнение, не поиск

Coherence score сам по себе не интерпретируется. Один запрос даёт число, осмысленное только в сопоставлении с другим. Канонический способ использования системы — `compare(A, B)`: запустить два reason'а и сравнить score. Даже одиночный запрос «?Сократ смертен» неявно — это сравнение с альтернативой «Сократ бессмертен». NSAI по своей сути дуальный инструмент: компаратор, не оракул.

## Тождество не вычисляется

Нет операции «разрешения тождества». К лемме в контексте подтягиваются ассоциации, обход естественным образом выбирает те, что согласованы с окружающим сигналом. «Лебедь летел» и «лебедь в озере» — две активации одной леммы с разными контекстными окрестностями; граф читает их как различные без явной метки тождества.

## Конфликт — не операция системы

Каждый факт получает свою согласованность независимо от других фактов о том же объекте. Если два утверждения об одном объекте оба дают высокий score в `reason()`, это *tension* — он всплывает как часть выхода рассуждения, не обнаруживается фоновым сканером. Tension существует только как свойство момента: сигнал должен задеть обе ветви, чтобы tension стал наблюдаемым. Альтернативы в графе, не активированные текущим сигналом, не имеют значения и не преследуются.

## Что хранится

Одна структура: `(предикат, [участники])`. Поле типа — фильтр для запросов, не архитектурное различие:

- `relation` → `(мыть, [мама, рама])`
- `entity` → `(яблоко, [])`
- `observation` → `('', [мыть, мама, рама])` — все леммы одного предложения, записываются для каждого `perceive`-вызова

Запись observation — то, что делает возможным joint-activation scoring. Без неё framing не имеет способа узнать, что несколько лемм наблюдались *вместе*, а не просто присутствуют в графе в разное время.

Args в `relation` идут в ролевом порядке (`nsubj → nsubj:pass → obj → iobj → obl → xcomp`), не по поверхностной позиции. Это даёт стабильный канонический ключ: «Кот ест рыбу» и «Рыбу ест кот» дают одну и ту же запись.

Предикат индексируется наравне с аргументами. Привилегированного пути для предикатов нет — они полноправные узлы графа.

Никаких метаданных, статусов, весов, confidence scores. `INSERT OR IGNORE` на дублях: 100 одинаковых подтверждений = 1 ребро. Открытый мир: отсутствие записи означает «не наблюдалось», не «ложно».

**Отрицательные универсальные утверждения не записываются.** «На столе ничего» — не факт, а бесконечный список негативов. Граф хранит наблюдения.

**Выводы не записываются.** Результаты рассуждения живут в оперативной памяти на время одного запроса и пересчитываются каждый раз, когда тот же вопрос задаётся снова.

## Частицы не получают спецобработки

Частица за которой следует предикат образует *составную лемму*. `не работает` становится одним узлом `не_работать`. `вряд ли работает` — `вряд_ли_работать`. Нет префикса отрицания, нет специального пути для `не`. Граф трактует `работать` и `не_работать` как два разных узла, которые могут разделять контекстные окрестности в одних местах и расходиться в других. Оппозиция эмерджентна, не закодирована.

## Оперативная память

Один уровень: сессионный кэш лемм недавних пользовательских сигналов. Кэш биасит стартовый фронт `floor_traversal` — тот же запрос, заданный в более насыщенном разговорном контексте, активирует более широкое семя.

Стека активных ассоциаций нет. В раннем дизайне был стек на 7 слотов, но слоты накапливали состояние, которое никем не читалось; убран вместе с линейным парсером.

## Self и мир

`Self = глобальная память; мир = входной сигнал.` Граница не требует явной метки. Мета-рассуждение возникает, когда тот же механизм обхода работает над фактами из собственного графа агента, и сам граф становится объектом инспекции.

## Curiosity

Цель автономного агента — **устранение альтернатив в собственном графе**, не максимизация знания, не исследование вовне. Агент действует, когда есть конкретная неоднозначность для разрешения, и отдыхает, когда граф устойчив.

Curiosity — выход `reason()`, не фоновое сканирование. Три типа сигналов, все — поверхностные проявления одного механизма (недостаточная или раздвоенная согласованность в обходе):
- **Дыра** — транзитивный глагол без объекта. Транзитивность читается через `pymorphy3.grammemes` единым хелпером `_is_transitive_verb`. Framing подсказывает кандидатов.
- **Tension** — два факта с полным покрытием сигнала, разделяющие одного участника, но непересекающиеся в остальных.
- **Низкая согласованность** — обход не накапливает достаточной поддержки ни в одном направлении.

Открытые вопросы ранжируются по `cascade_score` — масштабу последующей работы, которую разрешение разблокирует.

Автономная итерация: tensions из одного `reason()` становятся следующим сигналом. Цикл бежит над собственными выходами, не над графом.

## Отвергнутые решения

Каждый пункт ниже был рассмотрен и отвергнут, потому что дублирует то, что уже производит базовый цикл:

- **Эпистемические метки источника** (1-е/2-е/3-е лицо provenance). Self = память; мир = сигнал. Граница следует из того, откуда пришли данные, не из метки.
- **`[UNKNOWN_NP]`, omission-типы в БД, `group_id`.** Недосказанность — свойство момента анализа, не самого факта.
- **Таблицы `identity_link` / `distinct_link`.** Тождество эмерджентно из пересечения ассоциаций; хранить его избыточно.
- **Сжатие как операция записи.** Укорачивание пути — эффект чтения, не модификации.
- **Плотность как фильтр доверия.** Согласованность — критерий обхода, не отдельный фильтрующий слой.
- **Веса на рёбрах, счётчики подтверждений.** Граф без весов. Счёты возникают внутри обхода и исчезают, когда тот заканчивается.
- **Обнаружение конфликта как отдельный механизм.** Конфликты — внешние наблюдения над независимо посчитанными фактами.
- **Префикс `НЕ_` как специальный маркер отрицания.** Заменён общим правилом particle-lemma compounding.
- **Отрицательные универсальные факты в графе.** «X не существует» не хранится; отсутствие — не наблюдается, не утверждается.
- **`template.json` как источник назначения ролей.** Роли приходят из UD-отношений natasha. Файл больше не загружается.
- **Линейный позиционный парсер** (`.split()` или окно токенов). Свободный порядок слов в русском требует структурного анализа через дерево зависимостей.
- **`observation` как отдельный архитектурный тип факта.** Это `(предикат='', [все_леммы])` — частный случай общей структуры факта, не отдельная сущность.
- **`find_chains` как выделенный алгоритм.** Bridge-леммы — наблюдаемое свойство обхода. Текущая реализация всё ещё явная; цель — убрать, когда обход станет достаточно полным.
- **Фоновый `scan_curiosity` для tensions.** Tensions возникают внутри `reason()`, не из сканирования графа без сигнала.
- **Coherence score как сумма счётов отдельных лемм.** Популярная лемма тащила score даже без совместного наблюдения. Score теперь считает только факты с полным покрытием сигнала.
- **Сила цепочки (`a_facts × b_facts`) как часть score.** Произведение растёт от популярности bridge-узла, что тащит score популярных хабов в графе. Перенесено в диагностику моста; в общий `coherence_score` входит только число цепочек.
- **Сортировка args по поверхностной позиции.** Создавала разные args-списки для семантически идентичных предложений. Заменена ролевым приоритетом, дедупликация работает без отдельного механизма.
- **Расхождение определения транзитивности между путями.** `analyze` и `scan_curiosity` использовали разные критерии. Заменено единым хелпером `_is_transitive_verb`. Один архитектурный факт — одна точка реализации.

## Парсер

Парсер — сенсорный слой, не ядро. Маппит русский текст в узлы графа из дерева зависимостей natasha, с pymorphy3 для лемматизации (направляемой UD-тегом от natasha).

Извлечение фактов по UD-отношениям:

| UD relation | Что генерирует |
|---|---|
| VERB + nsubj/obj/iobj/obl/xcomp | relation: предикат → [args] |
| amod (ADJ ← NOUN) | relation: модификатор → [head] |
| ADJ как root + nsubj | relation: ADJ-предикат → [subject] |
| NOUN с case-ADP-ребёнком (head тоже NOUN) | relation: предлог → [head, dep] |
| conj/appos/nmod-сиблинги | присоединяются в args-список |
| acl:relcl | антецедент через head; если head — ADJ-предикат, разыменовывается через его nsubj |
| advmod + PART | составная лемма: `не_спать`, `вряд_ли_знать` |
| Литералы (числа, латиница) | entity, лемма = поверхностная форма |
| Несвязанные NOUN/PROPN | entity |

Args идут в ролевом порядке (`nsubj → nsubj:pass → obj → iobj → obl → xcomp`), внутри роли — в порядке возврата natasha с присоединёнными conj/appos/nmod-сиблингами сразу после головы своей роли. Поверхностная позиция не используется.

**Транзитивность.** `Case=Acc` у объекта означает транзитивность для окружающего глагола. Для детекции дыр (везде, где требуется) система запрашивает `pymorphy3.grammemes` через единый хелпер `_is_transitive_verb`. Словарный lookup надёжен для одиночной леммы.

**Составные леммы.** Частица + головной глагол сливаются через `advmod+PART`. Отрицание, модальность, эмфаза — все обрабатываются одним и тем же общим правилом, без специальных случаев. Разные составные леммы попадают в разные топологические окрестности через обход.

**Fallback.** Natasha иногда мисклассифицирует очень короткие изолированные предложения (например `мама мыла раму` → `мама` как root, `мыла` как nmod). Признак: NOUN-root в сочетании с осиротевшим VERB. В этом случае парсер переключается на линейный проход: первый VERB как предикат, все NOUN/PROPN/PRON/ADJ как аргументы.

## Статус

Черновик v0.1. Архитектура выше устойчива на уровне, при котором дальнейшие изменения взвешиваются осторожно относительно принципа отвергнутых решений. Известные пробелы для v0.2:

- Знаки препинания, операторы и числа как полноправные узлы графа — открывает архитектуру для нелингвистических входов.
- Удаление явного `find_chains`, когда подсчёт обхода станет достаточно полным, чтобы bridge-леммы всплывали без выделенного прохода. Альтернатива — per-query traversal с пересечением множеств достижимости.
- Сильно вложенные относительные клаузы natasha разбирает с ошибками (head иногда указывает на сам узел). Документировано как граница системы, не баг.
